# CLI Report Generation Design

**Status**: ✅ **APPROVED** - Aligned with Spectify Philosophy  
**Version**: 1.0  
**Date**: 2025-12-22  
**Author**: System Design  
**Target**: v0.8.0 CLI  

---

## Philosophy: Ephemeral by Default, Persistent by Choice

Spectify is a **lightweight, in-memory, hyperfast linting system** built for instant feedback. Results should be:
- **Ephemeral by default** - No automatic persistence, no cleanup needed
- **Persistent by choice** - Users decide what to save, where, and for how long
- **Version control friendly** - Reports can be committed, diffed, reviewed
- **CI/CD ready** - Generate artifacts that fit existing workflows

This aligns with Spectify's core design principles:
- Zero-copy architecture
- Worker-per-ruleset for speed
- Document affinity caching
- **No database creep**

---

## Problem Statement

Users need to:
1. **Save lint results** for later review, comparison, or compliance
2. **Share results** with team members or across systems
3. **Track changes** over time (before/after refactoring)
4. **Integrate with CI/CD** and store artifacts
5. **Generate human-readable summaries** for stakeholders

**Current limitations:**
- CLI only shows results in terminal (ephemeral)
- No way to save results except manual copy-paste
- Server stores results in memory (lost on restart)
- History tracking adds complexity without solving persistence

---

## Proposed Solution: Report Generation

### Core Concept

Add `--output` flag to `results` command to save **SARIF reports** generated by the server.

**Architecture**: Server-side SARIF generation
- Server generates SARIF 2.1.0 via `POST /lint/{jobId}/reports/generate`
- CLI requests SARIF and saves to file
- Single source of truth (server has lint results)
- All clients (CLI, MCP, API consumers) get consistent SARIF

```bash
# Workflow
spectify lint api.yaml --ruleset pubhub
# → Job submitted: job-abc123

# View results in terminal (ephemeral)
spectify results job-abc123

# Save SARIF report (persistent)
spectify results job-abc123 --output report.sarif

# Output SARIF to stdout (pipe to tools)
spectify results job-abc123 --format sarif | jq '.runs[0].results'
```

### Design Principles

1. **User controls persistence** - No automatic file generation
2. **Server-side generation** - Server transforms LintJobResult → SARIF
3. **SARIF 2.1.0 only (Phase 1)** - Industry standard, defer other formats
4. **Single source of truth** - Server has results, generates canonical SARIF
5. **API-first** - All clients (CLI, MCP, direct API) get consistent reports
6. **Idempotent** - Same jobId → same SARIF output
7. **Offline-capable** - Reports are self-contained
8. **`results` command integration** - Natural fit (per-jobId operation)

## Report Formats

### Architecture: Server-Side SARIF Generation

```
┌─────────────────────────┐
│  Spectify Server        │
│                         │
│  LintJobResult (memory) │
│         ↓               │
│  POST /lint/{id}/reports/generate │
│         ↓               │
│  SARIF Builder          │
│         ↓               │
│  SARIF 2.1.0 (JSON)     │
└─────────────────────────┘
         ↓
    ┌────┴────┬─────────────┐
    ↓         ↓             ↓
  CLI      MCP Tools    Direct API
 (saves)  (consume)     (consume)
```

**Why server-side generation?**
- Results already in server (LintJobResult in memory/Redis)
- Single SARIF implementation (DRY principle)
- All clients get consistent output
- Server-side caching possible
- API consumers benefit immediately
- CLI stays lightweight (thin client)

**Why SARIF-only Phase 1?**
- Industry standard (GitHub Code Scanning, VS Code, IDEs)
- Immediate tool integration value
- Future-proof for LLM/agent workflows
- Defer other formats until proven needed

## CLI Interface

### Basic Usage (Phase 1)

```bash
# Step 1: Submit lint job
spectify lint api.yaml --ruleset pubhub
# → Job submitted: job-abc123

# Step 2: View results in terminal (default)
spectify results job-abc123

# Step 3: Save SARIF report to file
spectify results job-abc123 --output report.sarif

# Or output to stdout (pipe to other tools)
spectify results job-abc123 --format sarif > report.sarif
spectify results job-abc123 --format sarif | jq '.runs[0].results'

# Explicit format flag (future: json, html, md)
spectify results job-abc123 --output report --format sarif
```

### Advanced Options (Phase 1)

```bash
# Timestamp in filename
spectify results job-abc123 \
  --output "report-$(date +%Y%m%d-%H%M%S).sarif"

# Use with latest job
spectify results --latest --output report.sarif

# Pipe to GitHub Code Scanning upload
spectify results job-abc123 --format sarif | \
  gh api repos/:owner/:repo/code-scanning/sarifs \
    --input - -f commit_sha="$GITHUB_SHA"
```

### Integration Examples

```bash
# CI/CD: Lint + save SARIF + upload to GitHub
JOB_ID=$(spectify lint api.yaml --ruleset pubhub --format json | jq -r '.jobId')
spectify results $JOB_ID --output report.sarif

# Upload to GitHub Code Scanning
gh api --method POST /repos/:owner/:repo/code-scanning/sarifs \
  -f sarif=@report.sarif -f commit_sha="$GITHUB_SHA"

# VS Code: Generate SARIF for Problems panel
spectify results $JOB_ID --output .vscode/spectify-results.sarif

# Direct pipe to analysis tools
spectify results $JOB_ID --format sarif | your-analyzer-tool
```

## Implementation Plan

### Phase 1: Server-Side SARIF Generation (v0.7.0 Server + v0.8.0 CLI)

**Scope:** SARIF 2.1.0 only, server-generated, CLI file saving

**Goal:** Server generates spec-compliant SARIF, CLI saves to file

#### Server Implementation (v0.7.0)

1. **SARIF Builder (src/formatters/sarif-builder.ts)**
   ```typescript
   // Server-side: Transforms LintJobResult → SARIF 2.1.0
   export class SarifBuilder {
     buildSarif(result: LintJobResult): SarifReport {
       return {
         $schema: 'https://json.schemastore.org/sarif-2.1.0.json',
         version: '2.1.0',
         runs: [this.buildRun(result)]
       };
     }
     
     private buildRun(result: LintJobResult): SarifRun {
       return {
         tool: this.buildTool(result.rulesetMetadata),
         artifacts: this.buildArtifacts(result.documentMetadata),
         results: this.buildResults(result.issues),
         properties: this.buildProperties(result)
       };
     }
     
     private generateFingerprints(issue: LintIssue): Record<string, string>;
   }
   ```

2. **API Endpoint (src/server.ts)**
   ```typescript
   // POST /lint/:jobId/reports/generate
   app.post('/lint/:jobId/reports/generate', async (request, reply) => {
     const { jobId } = request.params;
     const { format } = request.body;  // 'sarif' for Phase 1
     
     if (format !== 'sarif') {
       return reply.code(400).send({ error: 'Only SARIF format supported' });
     }
     
     const result = await storage.getJob(jobId);
     if (!result || result.status !== 'completed') {
       return reply.code(404).send({ error: 'Job not found or not completed' });
     }
     
     const sarif = sarifBuilder.buildSarif(result);
     return reply.send(sarif);
   });
   ```

#### CLI Implementation (v0.8.0)

1. **Results Command Enhancement (src/cli/commands/results.ts)**
   ```typescript
   // Add --output and --format flags to results command
   program
     .command('results')
     .argument('<job-id>', 'Job ID to retrieve results for')
     .option('--output <path>', 'Save report to file')
     .option('--format <format>', 'Output format (sarif)', 'sarif')
     .action(async (jobId, options) => {
       if (options.output || options.format) {
         // Request SARIF from server
         const sarif = await apiClient.generateReport(jobId, { format: 'sarif' });
         
         if (options.output) {
           await fs.writeFile(options.output, JSON.stringify(sarif, null, 2));
           console.log(`SARIF report saved to ${options.output}`);
         } else {
           // Output to stdout
           console.log(JSON.stringify(sarif, null, 2));
         }
       } else {
         // Default: show terminal formatted results
         const results = await apiClient.getResults(jobId);
         formatters.terminal(results);
       }
     });
   ```

2. **API Client (src/cli/api-client.ts)**
   ```typescript
   export class ApiClient {
     async generateReport(jobId: string, options: { format: 'sarif' }): Promise<SarifReport> {
       const response = await fetch(`${this.baseUrl}/lint/${jobId}/reports/generate`, {
         method: 'POST',
         headers: { 'Content-Type': 'application/json' },
         body: JSON.stringify(options)
       });
       
       if (!response.ok) {
         throw new Error(`Failed to generate report: ${response.statusText}`);
       }
       
       return response.json();
     }
   }
   ```

## Report Data Structure

### Core Report Schema

```typescript
interface SpectifyReport {
  spectify: {
    version: string;           // "0.6.1-cli0.8.0"
    generated_at: string;      // ISO 8601 timestamp
    generator: string;         // "cli" | "api" | "mcp"
  };
  
  metadata: {
    document: DocumentMetadata;
    execution: ExecutionMetadata;
    environment?: EnvironmentMetadata;
  };
  
  summary: {
    total_issues: number;
    by_severity: Record<Severity, number>;
    by_category?: Record<string, number>;
    passed: boolean;
    score?: number;              // 0-100, optional
  };
  
  issues: Issue[];
  
  rules_executed?: {
    total: number;
    passed: number;
    failed: number;
    skipped?: number;
  };
}

interface DocumentMetadata {
  id: string;
  name?: string;
  path: string;
  size_bytes: number;
  format: string;               // "openapi-3.0.2", "swagger-2.0", etc.
  hash?: string;                // Document content hash
}

interface ExecutionMetadata {
  job_id: string;
  ruleset: string;
  ruleset_version: string;
  started_at: string;
  completed_at: string;
  duration_ms: number;
  server_url?: string;
}

interface EnvironmentMetadata {
  hostname?: string;
  user?: string;
  cwd?: string;
  ci?: boolean;
  git?: {
    branch?: string;
    commit?: string;
    dirty?: boolean;
  };
}

interface Issue {
  code: string;
  message: string;
  severity: 'error' | 'warn' | 'info' | 'hint';
  path: (string | number)[];
  range?: {
    start: { line: number; column: number };
    end: { line: number; column: number };
  };
  category?: string;
  documentation?: string;
}
```

## History Management Strategy

With report generation, **history becomes simple and ephemeral**:

### Current Session Only

```bash
spectify history
# Shows jobs from current server instance only
# Auto-clears on server restart
# No stale entries, no cleanup needed
```

### Implementation

```typescript
// CLI history management (simplified)
class HistoryManager {
  // History tied to server uptime, not execution_id
  async getHistory(): Promise<HistoryEntry[]> {
    const entries = this.loadHistory();
    
    // Simple staleness check: can we reach the server?
    const serverReachable = await this.pingServer();
    if (!serverReachable) {
      // Server down/restarted - clear history
      this.clearHistory();
      return [];
    }
    
    return entries;
  }
  
  // No complex validation, no execution IDs, no checksums
  async addEntry(job: LintJob) {
    const entry = {
      jobId: job.id,
      timestamp: new Date().toISOString(),
      documentPath: job.documentPath,
      ruleset: job.rulesetName
    };
    this.appendHistory(entry);
  }
}
```

Report generation overhead:

  - HTML: +50ms (template + embedded assets)
  - Markdown: +15ms (template rendering)

Total impact: < 5% in worst case

async generateLargeReport(results: LintJobResult, outputPath: string) {
  const stream = fs.createWriteStream(outputPath);
  
  stream.write('{\n');
  stream.write('"spectify": ' + JSON.stringify(results.metadata) + ',\n');
  stream.write('"issues": [\n');
  
  for (const issue of results.issues) {

  }
  
  stream.write(']\n}');
  stream.end();
}

## Open Questions

1. **SARIF Caching**: Should server cache generated SARIF reports?
   - **Consideration**: Idempotent (same jobId → same SARIF), so caching makes sense
   - **Implementation**: Cache key = `{jobId}:sarif`, TTL = job result TTL
   - **Decision**: Yes, cache in Phase 1 ✅

2. **SARIF Schema Version**: Lock to 2.1.0 or support configurable?
   - **Decision**: Lock to 2.1.0 (latest, widely supported) ✅
   - Future: Add version parameter if 2.2.0 released

3. **Fingerprint Schemes**: Which schemes to implement in Phase 1?
   - **Decision**: `primaryLocationLineHash` only (simple, sufficient) ✅
   - Future: Add semantic hashing if deduplication needs arise

4. **Other Formats**: When to implement JSON/HTML/Markdown?
   - **Decision**: Deferred until Phase 1 adoption measured ✅
   - Evaluation criteria: User requests, tool gaps, stakeholder needs

5. **Server vs Client**: If other formats needed, where to implement?
   - **Options**: Server (consistent), Client (flexible), External tools (modular)
   - **Decision**: Deferred, evaluate per format based on Phase 1 feedback

## Conclusion

**Server-side SARIF generation** is the **correct solution** for Spectify. It:
- ✅ Preserves lightweight, ephemeral nature (no automatic persistence)
- ✅ Gives users control over persistence (explicit --output flag)
- ✅ **Single source of truth** (server has results, generates SARIF)
- ✅ **All clients benefit** (CLI, MCP, direct API consumers)
- ✅ **Immediate tool integration** (GitHub Code Scanning, VS Code, IDEs)
- ✅ **Industry standard** (SARIF 2.1.0 widely adopted)
- ✅ **Future-proof for LLM/agent workflows** (confidence scores, fingerprints)
- ✅ **API-first design** (CLI thin, server smart)
- ✅ Avoids database creep and state management complexity
- ✅ Scales from quick local checks to enterprise compliance
- ✅ **DRY principle** (one SARIF implementation, not duplicated)

**Why server-side wins:**
1. **Results already there**: LintJobResult in server memory/Redis
2. **Single implementation**: No duplication across clients
3. **Consistent output**: All consumers get same SARIF
4. **Caching opportunity**: Server can cache generated reports
5. **API-first**: Programmatic consumers benefit immediately

**Why SARIF-only Phase 1 wins:**
1. **Immediate value**: GitHub integration from day one
2. **Industry standard**: Tooling ecosystem ready
3. **Rich metadata**: Rule catalogs, fingerprints, confidence
4. **Defer complexity**: Other formats only if proven needed
5. **Fast to ship**: Single format, well-specified

**Next Steps:**
1. ✅ Design approved (server-side SARIF via POST)
2. Implement server SARIF builder for v0.7.0
3. Implement CLI --output flag on `results` command for v0.8.0
4. Validate SARIF against official JSON Schema
5. Test GitHub Code Scanning upload
6. Ship and gather user feedback
7. Evaluate other formats based on adoption and requests
